# Multi Goal Rules

Этот документ описывает, как бот и backend должны работать, если у одного пользователя несколько целей одновременно.

---

## 1. Общий принцип

У одного пользователя может быть несколько целей.
Каждая цель должна существовать как отдельный независимый трек внутри системы.

Это значит, что у каждой цели отдельно хранятся:
- контекст цели
- план
- шаги плана
- daily check-ins
- proof-файлы
- прогресс
- weekly summary
- AI-контекст
- состояние сценария бота по этой цели

Цели не должны смешиваться между собой.

---

## 2. Один чат, но разные цели

У Telegram-бота один чат с пользователем.
Новые цели не создают новый чат и не создают новую вкладку Telegram.

Вместо этого внутри одного чата существует:
- список целей пользователя
- одна выбранная активная цель для текущего контекста общения

---

## 3. Active Goal

В каждый момент времени у пользователя есть `active_goal_id`.

`active_goal_id` означает:
- какая цель сейчас выбрана в чате
- к какой цели по умолчанию относятся сообщения пользователя
- какой goal context показывается в интерфейсе

Важно:
`active_goal_id` — это только текущий chat-context.
Это не значит, что у пользователя только одна цель в системе.

---

## 4. Отдельный контекст для каждой цели

Каждая цель должна иметь собственный набор данных.

Для каждой цели отдельно существуют:
- `goal_id`
- `goal_session`
- `plan`
- `plan_steps`
- `daily_checkins`
- `proof_files`
- `goal_progress`
- `weekly_summary`

Backend не должен работать с логикой "текущие данные пользователя вообще".
Backend должен работать по схеме:
- пользователь
- конкретная цель
- действие по этой цели

То есть почти каждое действие должно быть привязано к `goal_id`.

---

## 5. Goal Session

У каждой цели должна быть своя `goal_session`.

`goal_session` хранит:
- state
- substate
- context_json
- updated_at

Это нужно, чтобы две цели одного пользователя не путались между собой.

Пример:
- цель "Похудение" может быть в состоянии `active_execution`
- цель "Гитара" может быть в состоянии `awaiting_plan_review`

Обе цели при этом существуют одновременно и не конфликтуют.

---

## 6. User Chat Context

Отдельно существует глобальный chat context пользователя.

Он хранит:
- `user_id`
- `active_goal_id`
- `last_selected_goal_id`
- глобальный state чата при необходимости
- updated_at

Это нужно только для навигации внутри одного чата.

---

## 7. Правило маршрутизации сообщений

### 7.1. Если пользователь нажал кнопку
Любое действие, вызванное кнопкой, должно по возможности содержать `goal_id`.

Например:
- открыть цель
- показать задачи на сегодня
- отправить отчет
- принять план
- запросить корректировку плана

В таких случаях backend не должен угадывать цель — она уже явно указана.

### 7.2. Если пользователь отправил обычный текст
Если текст пришел без явного `goal_id`, используются такие правила:

1. если у пользователя одна активная цель — сообщение относится к ней
2. если у пользователя несколько целей, но есть `active_goal_id` и текущий сценарий по ней ожидает ответ — сообщение относится к этой цели
3. если контекст неоднозначный — бот должен попросить выбрать цель

### 7.3. Если сообщение нельзя уверенно привязать к цели
Backend или бот не должны автоматически угадывать цель с низкой уверенностью.
В этом случае бот должен попросить пользователя выбрать цель явно.

---

## 8. Правило для кнопок и UI

Все важные действия должны быть goal-aware.

Это значит:
- в callback кнопок должен передаваться `goal_id`
- при показе задач, прогресса и summary должна явно отображаться цель
- при отправке сообщений бот должен показывать название цели

Пример:
- "Цель: Похудение"
- "Цель: Гитара"

Это уменьшает путаницу для пользователя.

---

## 9. План и выполнение по нескольким целям

Один пользователь может иметь несколько целей.
У каждой цели может быть:
- свой план
- свой reminder
- свой daily check-in
- свой weekly summary

Check-in по одной цели не должен влиять на check-in по другой цели.

Proof по одной цели не должен быть доступен как proof по другой цели без явной привязки.

---

## 10. Прогресс по нескольким целям

Прогресс считается отдельно по каждой цели.

Нельзя считать один общий прогресс на пользователя по всем целям сразу, если это не отдельная аналитическая функция.

Для каждой цели отдельно считаются:
- progress_percent
- current_streak
- completed_steps_count
- planned_steps_count
- missed_days_count
- weekly summary

---

## 11. Reminders по нескольким целям

Reminder должен быть привязан к конкретной цели.

Один пользователь может иметь:
- разные reminders для разных целей
или
- reminders только для части целей

Backend должен поддерживать reminder на уровне `goal_id`, а не только на уровне `user_id`.

---

## 12. AI Context Rules

AI не должен получать весь общий чат пользователя, если у пользователя несколько целей.

При любом AI-вызове контекст должен собираться отдельно по одной цели.

В AI context можно включать:
- описание конкретной цели
- данные профилирования по этой цели
- time budget по этой цели
- текущий план по этой цели
- последние check-ins по этой цели
- последние результаты проверки proof по этой цели
- прогресс по этой цели

В AI context нельзя автоматически подмешивать другую цель пользователя, если это не специальный сценарий.

---

## 13. Переключение между целями

Когда пользователь переключает активную цель:
- обновляется `active_goal_id`
- загружается `goal_session` новой цели
- бот начинает работать в контексте новой цели
- бот должен явно сообщить, что активная цель изменена

Пример:
- "Переключил тебя на цель: Гитара"

---

## 14. Сброс и изоляция состояния

При переключении активной цели важно не переносить случайно локальное состояние одной цели в другую.

То есть:
- если по цели A бот ждет proof
- это не значит, что по цели B он тоже ждет proof

Состояние каждой цели изолировано в `goal_session`.

---

## 15. MVP-правила

Для первой версии продукта принимаются такие правила:

- у пользователя может быть несколько целей
- в интерфейсе чата всегда есть одна выбранная активная цель
- у каждой цели отдельные plan, check-in, proof, progress и goal_session
- все важные кнопки и действия должны передавать `goal_id`
- если текст нельзя надежно привязать к цели, бот просит выбрать цель
- AI работает только с контекстом одной выбранной цели за раз

---

## 16. Ошибки, которых нужно избегать

Нельзя:
- использовать один общий conversation state на все цели пользователя
- передавать все цели пользователя в один AI prompt без необходимости
- сохранять check-in без `goal_id`
- сохранять proof без связи с `goal_id` и `check-in`
- пытаться угадывать цель по свободному тексту, если уверенности нет
- показывать прогресс без указания, к какой цели он относится

---

## 17. Практический вывод для backend

Почти каждая backend-команда должна принимать:
- `user_id`
- `goal_id`

И почти каждый объект в бизнес-логике должен быть связан с `goal_id`.

Это главное правило multi-goal режима.
