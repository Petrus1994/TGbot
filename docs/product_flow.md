# Product Flow

Этот документ описывает основной пользовательский сценарий работы Telegram-бота.

---

## 1. Первый запуск

1. Пользователь нажимает `/start`.
2. Система проверяет, существует ли пользователь в базе.
3. Если пользователя еще нет, backend создает нового пользователя.
4. Бот приветствует пользователя и объясняет, что умеет:
   - помочь сформулировать цель
   - собрать контекст
   - составить персональный план
   - сопровождать выполнение
   - проверять proof выполнения
   - показывать прогресс

---

## 2. Создание первой цели

1. Бот спрашивает: какую цель пользователь хочет достичь.
2. Пользователь отправляет описание цели в свободной форме.
3. Backend создает `goal` в статусе `draft`.
4. Бот переходит к этапу уточнения цели.

Примеры целей:
- научиться играть на гитаре
- похудеть на 8 кг
- выучить английский до разговорного уровня
- начать бегать 3 раза в неделю

---

## 3. Уточнение цели и профилирование

После того как цель получена, бот задает уточняющие вопросы.

Цель этапа:
- понять текущий уровень пользователя
- понять ограничения
- понять доступные ресурсы
- понять мотивацию
- понять желаемый темп

Примеры вопросов:
- какой у тебя сейчас уровень в этой деятельности?
- есть ли ограничения по времени?
- есть ли инструменты, материалы или оборудование?
- есть ли физические, финансовые или бытовые ограничения?
- почему для тебя важна эта цель?
- есть ли дедлайн?
- сколько времени в день или в неделю ты готов уделять?

На этом этапе:
1. backend сохраняет ответы пользователя
2. обновляет `goal context`
3. отслеживает прогресс профилирования
4. определяет, достаточно ли данных для генерации плана

Когда обязательные данные собраны:
- цель переходит в статус `profiling` -> `planning`

---

## 4. Генерация плана

1. Backend собирает контекст по конкретной цели.
2. Формируется AI-запрос только по этой цели.
3. AI генерирует персональный план.
4. Backend сохраняет план и его шаги.
5. План получает статус `ready_for_review`.
6. Бот показывает пользователю план.

План должен содержать:
- понятную структуру
- конкретные шаги
- регулярность выполнения
- ожидаемый формат подтверждения
- нагрузку, соответствующую возможностям пользователя

---

## 5. Согласование плана

После показа плана пользователь может:

### Вариант A. Принять план
1. Пользователь нажимает принять план.
2. План получает статус `accepted`.
3. Цель получает статус `active`.
4. Начинается выполнение.

### Вариант B. Попросить изменить план
1. Пользователь просит скорректировать план.
2. Бот спрашивает, что именно нужно изменить.
3. Пользователь отправляет комментарий.
4. Backend запускает пересборку плана.
5. Создается новая версия плана.
6. Пользователю снова показывается обновленный план.

---

## 6. Начало выполнения цели

После принятия плана:

1. Цель становится активной.
2. Для цели включается рабочий сценарий выполнения.
3. Для каждого дня или периода формируются задачи.
4. При необходимости включаются reminders.
5. Пользователь может открыть задачи на сегодня.

---

## 7. Ежедневное выполнение

Для активной цели пользователь должен выполнять шаги плана.

Пользователь может:
- открыть задачи на сегодня
- прочитать, что именно нужно сделать
- отправить текстовый отчет
- отправить фото или скриншот как proof выполнения

Backend должен:
- создать `daily_checkin`
- привязать check-in к `goal_id`
- сохранить статус выполнения шагов
- сохранить текстовые и файловые подтверждения

---

## 8. Proof validation

Если шаг требует подтверждения:

1. Пользователь отправляет фото или скриншот.
2. Backend сохраняет файл в storage.
3. Файл прикрепляется к check-in и конкретному шагу.
4. Запускается проверка proof.
5. AI или другая логика анализа определяет результат.

Возможные результаты:
- `passed` — подтверждение принято
- `failed` — подтверждение не принято
- `uncertain` — система не уверена

Если proof принят:
- шаг засчитывается
- прогресс пересчитывается

Если proof отклонен:
- бот просит прислать более четкое подтверждение

---

## 9. Ежедневный прогресс

Каждый день система должна:
1. пересчитать прогресс по активной цели
2. определить, сколько шагов выполнено
3. определить, сколько еще осталось
4. отправить пользователю краткое сообщение с прогрессом

Сообщение должно содержать:
- текущий процент прогресса
- результат за день
- краткую мотивацию или рекомендацию

---

## 10. Weekly summary

В конце каждой недели система должна:
1. собрать данные за неделю
2. посчитать выполненные и пропущенные шаги
3. посчитать изменение прогресса
4. сформировать summary
5. отправить пользователю сообщение

Summary должен содержать:
- сколько было задач
- сколько выполнено
- сколько пропущено
- как изменился прогресс
- мотивирующее сообщение

Тон summary может быть:
- supportive
- strict_supportive

---

## 11. Работа с несколькими целями

У одного пользователя может быть несколько целей.

Важно:
- новая цель не создает новый Telegram-чат
- все цели живут внутри одного чата
- у каждой цели отдельный контекст, план, check-ins, proof, progress и goal session
- в каждый момент у пользователя есть `active_goal_id`

Это значит:
- бот всегда понимает, с какой целью он сейчас работает
- пользователь может переключаться между целями
- AI работает только с одной выбранной целью за раз

---

## 12. Переключение между целями

Если у пользователя несколько целей:

1. пользователь открывает список целей
2. выбирает нужную цель
3. backend обновляет `active_goal_id`
4. бот сообщает, какая цель теперь активна
5. дальнейшие действия относятся к новой активной цели

При этом:
- состояние одной цели не должно смешиваться с состоянием другой
- proof, check-ins и progress всегда привязаны к конкретному `goal_id`

---

## 13. Напоминания

Для каждой цели могут быть включены reminders.

Reminder должен учитывать:
- goal_id
- timezone пользователя
- локальное время пользователя
- статус цели

Напоминания могут использоваться для:
- напоминания о задачах на сегодня
- напоминания отправить check-in
- возврата пользователя после пропуска

---

## 14. Пропуски и возврат в выполнение

Если пользователь не отправил check-in:
1. система фиксирует пропущенный день
2. обновляет прогресс и streak
3. при необходимости отправляет мягкое или строго-мотивирующее сообщение
4. предлагает вернуться к выполнению

---

## 15. Завершение цели

Цель может завершиться тремя способами:

### completed
Пользователь достиг цели.

### paused
Пользователь временно остановил выполнение.

### cancelled
Пользователь отказался от цели или закрыл ее.

После завершения:
- цель больше не участвует в активных reminders
- progress сохраняется в истории
- weekly summaries по ней больше не отправляются

---

## 16. Общий принцип backend-логики

Почти каждое действие в системе должно быть связано с:
- `user_id`
- `goal_id`

Это нужно, чтобы:
- не путались несколько целей
- AI работал с правильным контекстом
- daily check-ins не смешивались
- proof относился к правильной цели
- прогресс считался корректно
