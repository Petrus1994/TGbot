# Stack Decisions

Этот документ фиксирует финальные технические решения по стеку проекта для MVP и дальнейшего роста.

---

## 1. Общий принцип

Для проекта выбирается стек, который:
- надежен
- масштабируем
- подходит для роста продукта
- не смешивает все критичные части в одну платформу
- дает хороший баланс между качеством инфраструктуры и скоростью команды из 2 человек

Итоговый стек:
- Telegram Bot API
- один backend-сервис
- PostgreSQL
- AWS S3 для хранения файлов
- отдельный worker / scheduler
- OpenAI API
- GitHub для кода и документации
- Railway для хостинга backend и worker

---

## 2. Telegram

### Выбор
Telegram Bot API

### Зачем
Telegram Bot API используется для:
- общения пользователя с ботом
- команд
- текста
- inline-кнопок
- отправки и получения фото
- reminders
- daily progress messages
- weekly summaries

---

## 3. Backend

### Выбор
Один основной backend-сервис

### Зачем
Backend отвечает за:
- бизнес-логику
- работу с PostgreSQL
- управление целями
- управление планами
- управление check-ins
- работу с proof validation flow
- интеграцию с OpenAI API
- работу с reminders
- event logging
- multi-goal isolation

### Принцип
Для MVP используется один backend, без микросервисов.
Это упрощает поддержку и ускоряет разработку.

---

## 4. Database

### Выбор
PostgreSQL

### Почему выбран PostgreSQL
PostgreSQL хорошо подходит для проекта, потому что:
- поддерживает сложные связи между сущностями
- надежен и зрелый
- хорошо подходит для transactional backend
- поддерживает JSONB
- хорошо подходит для статусов, событий, check-ins, reminders и версий планов

### Что хранится в PostgreSQL
- пользователи
- профили пользователей
- цели
- сессии целей
- chat context
- планы
- шаги планов
- daily check-ins
- step reports
- напоминания
- deliveries
- события
- логи исходящих сообщений
- AI run metadata
- метаданные файлов

### Примечание
Файлы не хранятся в PostgreSQL как binary/blob.
В PostgreSQL хранятся только ссылки и метаданные файлов.

---

## 5. File Storage

### Выбор
AWS S3

### Почему выбран AWS S3
AWS S3 выбран как основное хранилище файлов, потому что:
- это зрелое и очень надежное объектное хранилище
- S3 designed for 99.999999999% durability and 99.99% availability by default
- данные хранятся избыточно и устойчиво
- хорошо подходит для пользовательских изображений и proof-файлов
- легко масштабируется
- хорошо интегрируется с backend и стандартными SDK

### Что хранится в S3
- фотопруфы
- скриншоты
- другие пользовательские изображения

### Что хранится в базе вместо файлов
- file_id
- storage_key
- bucket_name
- mime_type
- size_bytes
- checksum
- owner user_id
- related goal_id / checkin_id

### Примечание
Cloudflare R2 рассматривается как сильная альтернативная опция по стоимости, особенно из-за отсутствия egress fees, но основной выбранный вариант — AWS S3.

## 6. AI Provider

### Выбор
OpenAI API

### Зачем
OpenAI API используется для:
- генерации плана по цели
- корректировки плана
- weekly summary
- motivational messages
- proof/image analysis
- определения, подтверждает ли изображение выполнение шага

### Принцип использования
AI должен работать только с контекстом одной конкретной цели.
Нельзя смешивать несколько целей пользователя в одном AI context без специальной необходимости.

---

## 7. Scheduler / Worker

### Выбор
Отдельный worker / scheduler process

### Почему нужен отдельный worker
Фоновые задачи не должны жить только внутри веб-процесса backend.
Отдельный worker нужен для:
- daily reminders
- daily progress jobs
- weekly summary jobs
- missed day detection
- reengagement messages
- асинхронных AI-задач
- retry logic при временных ошибках

### Принцип
Worker и backend логически разделены, даже если хостятся на одной платформе.

---

## 8. Hosting / Deploy

### Выбор
Railway

### Почему выбран Railway
Railway выбран как хостинг для backend и worker, потому что:
- быстрее и проще для команды из 2 человек, чем самостоятельная настройка AWS-инфраструктуры
- подходит для быстрого deploy backend-сервисов
- позволяет хостить web service и worker
- дает удобный developer workflow
- подходит для MVP и дальнейшего роста без лишней DevOps-нагрузки

### Что размещается на Railway
- backend service
- worker / scheduler
- environment variables
- deploy configuration

### Примечание
Railway выбран как лучший баланс между качеством и скоростью команды.
Если в будущем проект сильно вырастет по нагрузке или требованиям, можно будет отдельно мигрировать часть инфраструктуры.

## 9. Environments

Должны существовать как минимум 3 окружения:

### local
Для локальной разработки.

### staging
Для проверки изменений перед production.

### production
Для реальных пользователей.

### Принцип
Нельзя тестировать критичные изменения сразу на production.

---

## 10. Secrets / Environment Variables

Все секреты должны храниться отдельно от кода.

### Основные env variables
- TELEGRAM_BOT_TOKEN
- DATABASE_URL
- OPENAI_API_KEY
- AWS_ACCESS_KEY_ID
- AWS_SECRET_ACCESS_KEY
- AWS_REGION
- S3_BUCKET_NAME
- APP_BASE_URL
- INTERNAL_API_SECRET
- DEFAULT_TIMEZONE
- ENVIRONMENT

### Правила
- секреты нельзя коммитить в GitHub
- реальные значения не должны попадать в docs
- в репозитории должен быть `.env.example`

## 11. Observability

### Минимальные требования
- structured logs
- логирование входящих Telegram events
- логирование исходящих сообщений
- логирование reminder jobs
- логирование AI runs
- логирование proof validation flow
- логирование ошибок backend
- логирование retry attempts

### Зачем это нужно
Чтобы понимать:
- почему бот не ответил
- почему reminder не отправился
- почему proof не был засчитан
- где сломался flow пользователя
- какие AI-задачи завершились ошибкой

---

## 12. Security

### Минимальные требования
- пользователь должен иметь доступ только к своим целям
- пользователь должен видеть только свои check-ins
- пользователь должен видеть только свои файлы
- все команды должны проверять ownership
- Telegram events должны обрабатываться идемпотентно
- размер файлов должен быть ограничен
- storage должен быть private
- секреты должны храниться безопасно
- критичные endpoints должны быть защищены

### Дополнительно
Нужно проверять:
- user_id + goal_id ownership
- file ownership
- checkin ownership
- защиту от дублирующих inbound events

## 14. Почему не выбран all-in-one вариант

Проект сознательно не строится полностью на одной all-in-one платформе типа:
- база + storage + backend + scheduler в одном месте

Причина:
- важные части системы лучше разделить логически
- база и файловое хранилище должны быть надежными и независимыми
- это снижает риск сильной связанности инфраструктуры
- проекту проще расти и мигрировать по частям
